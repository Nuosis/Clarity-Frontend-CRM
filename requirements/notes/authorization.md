# Notes Feature - Authorization and Row-Level Security

## Overview

This document defines the authorization model and Row-Level Security (RLS) policies for the notes feature. The authorization strategy ensures that:

1. **Organization Isolation**: Users can only access notes within their organization
2. **Parent Entity Access Inheritance**: Note access is tied to parent entity (customer/project/task) access
3. **Simple Permissions**: All authenticated users in an organization have the same note permissions
4. **No Role-Based Restrictions**: Matches current FileMaker implementation (all users can create/view/edit/delete notes)

**Current FileMaker Behavior:**
- All users with FileMaker access can create notes
- All users can view all notes in the system (no restrictions)
- No update or delete operations implemented in current UI
- No organization scoping (single-tenant FileMaker environment)

**Supabase Migration Strategy:**
- Preserve simple permission model (all authenticated org users have full access)
- Add organization scoping for multi-tenancy
- Add parent entity access validation (can't create notes on entities you can't access)
- Enable update/delete operations for future UI implementation
- Use service role key for backend API operations (bypasses RLS)
- Use anon key for direct Supabase access from frontend (subject to RLS)

## Authentication Methods

### Backend API (HMAC Authentication)

**Authentication:** HMAC-SHA256 signature
**Header Format:** `Authorization: Bearer {signature}.{timestamp}`
**Usage:** All requests to `https://api.claritybusinesssolutions.ca/api/notes/*`

**Implementation:**
- Generated by: `src/services/dataService.js:generateBackendAuthHeader()`
- Secret key: `VITE_SECRET_KEY` environment variable
- Validates: Request integrity, timestamp freshness, prevents tampering

**Authorization:**
- Backend validates HMAC signature
- Backend extracts user context from request session/JWT
- Backend enforces organization scoping at application level
- Backend validates parent entity access before allowing note operations

**Code Reference:** `src/services/dataService.js:generateBackendAuthHeader()`

---

### Supabase Direct Access (JWT Authentication)

**Authentication:** Supabase JWT token
**Header Format:** `Authorization: Bearer {supabase_jwt_token}`
**Usage:** Direct Supabase client queries and realtime subscriptions

**JWT Claims Required:**
```json
{
  "sub": "user_uuid",
  "role": "authenticated",
  "aud": "authenticated",
  "user_metadata": {
    "organization_id": "org_uuid"
  }
}
```

**Authorization:**
- RLS policies automatically filter by `organization_id` from JWT
- All queries scoped to user's organization
- No additional authorization checks needed at application level

**Service Role Key:**
- **When to use:** Backend operations that need to bypass RLS (migrations, admin operations)
- **NOT for:** Frontend client access (security risk)
- **Environment variable:** `VITE_SUPABASE_SERVICE_ROLE_KEY`

**Anon Key:**
- **When to use:** Frontend Supabase client access, realtime subscriptions
- **Subject to:** All RLS policies
- **Environment variable:** `VITE_SUPABASE_ANON_KEY`

## RLS Helper Functions

### Organization Context Function

```sql
-- Get current user's organization ID from JWT claims
CREATE OR REPLACE FUNCTION auth.current_organization_id()
RETURNS UUID
LANGUAGE sql
STABLE
AS $$
  SELECT COALESCE(
    (current_setting('request.jwt.claims', true)::json->'user_metadata'->>'organization_id')::uuid,
    NULL
  );
$$;

COMMENT ON FUNCTION auth.current_organization_id() IS
  'Extracts organization_id from JWT user_metadata. Returns NULL if not authenticated or claim missing.';
```

**Usage in Policies:**
- All RLS policies use this function to determine user's organization
- Function is `STABLE` for query optimization (result cached within transaction)
- Returns `NULL` for unauthenticated requests (triggers policy denial)

---

### Parent Entity Access Validation Functions

```sql
-- Check if user has access to a customer
CREATE OR REPLACE FUNCTION auth.has_customer_access(p_customer_id UUID)
RETURNS BOOLEAN
LANGUAGE sql
STABLE SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.customers
    WHERE id = p_customer_id
    AND organization_id = auth.current_organization_id()
  );
$$;

COMMENT ON FUNCTION auth.has_customer_access(UUID) IS
  'Returns true if customer exists and belongs to user''s organization';

-- Check if user has access to a project
CREATE OR REPLACE FUNCTION auth.has_project_access(p_project_id UUID)
RETURNS BOOLEAN
LANGUAGE sql
STABLE SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.projects
    WHERE id = p_project_id
    AND organization_id = auth.current_organization_id()
  );
$$;

COMMENT ON FUNCTION auth.has_project_access(UUID) IS
  'Returns true if project exists and belongs to user''s organization';

-- Check if user has access to a task
-- Tasks don't have direct organization_id, check via project
CREATE OR REPLACE FUNCTION auth.has_task_access(p_task_id UUID)
RETURNS BOOLEAN
LANGUAGE sql
STABLE SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.tasks t
    JOIN public.projects p ON t.project_id = p.id
    WHERE t.id = p_task_id
    AND p.organization_id = auth.current_organization_id()
  );
$$;

COMMENT ON FUNCTION auth.has_task_access(UUID) IS
  'Returns true if task exists and its parent project belongs to user''s organization';
```

**Security Notes:**
- Functions use `SECURITY DEFINER` to bypass RLS when checking access
- This is necessary because RLS policies use these functions (avoiding circular dependency)
- Functions are safe because they only check organization membership, not perform operations

### SELECT Policy: View Notes

**Policy Name:** `notes_select_policy`

**Rule:** Authenticated users can view all notes in their organization

```sql
CREATE POLICY "notes_select_policy"
ON public.notes
FOR SELECT
TO authenticated
USING (
  organization_id = auth.current_organization_id()
);
```

**Rationale:**
- Matches current FileMaker behavior (all users see all notes)
- Simple organization scoping for multi-tenancy
- No role-based restrictions
- Query performance: Uses indexed `organization_id` column

**Example Queries:**
```javascript
// All queries automatically filtered by organization
const { data } = await supabase
  .from('notes')
  .select('*')
  .eq('project_id', projectId);
// Returns only notes where:
// - project_id = projectId
// - organization_id = user's organization (automatic via RLS)
```

---

### INSERT Policy: Create Notes

**Policy Name:** `notes_insert_policy`

**Rule:** Authenticated users can create notes in their organization on entities they can access

```sql
CREATE POLICY "notes_insert_policy"
ON public.notes
FOR INSERT
TO authenticated
WITH CHECK (
  -- Must be user's organization
  organization_id = auth.current_organization_id()

  -- Must have access to parent entity
  AND (
    (customer_id IS NOT NULL AND auth.has_customer_access(customer_id))
    OR
    (project_id IS NOT NULL AND auth.has_project_access(project_id))
    OR
    (task_id IS NOT NULL AND auth.has_task_access(task_id))
  )
);
```

**Validation Checks:**
1. **Organization Match**: Note's `organization_id` must match user's organization
2. **Parent Entity Access**: User must have access to the parent entity (customer/project/task)
3. **Parent Entity Existence**: Parent entity must exist and belong to user's organization

**Error Scenarios:**
- **Wrong Organization**: Attempting to create note with different `organization_id` → Policy violation
- **No Parent Access**: Attempting to create note on another org's project → Policy violation via `has_project_access()`
- **Invalid Parent**: Parent entity doesn't exist → Foreign key constraint violation (before RLS check)

**Frontend Implications:**
- Frontend should NOT manually set `organization_id` - backend derives it from parent entity
- Frontend validates parent entity exists before attempting creation
- Error handling: Policy violations return 403 Forbidden from Supabase

**Code Reference:**
- Validation: `src/services/noteService.js:11-17`
- Creation: `src/hooks/useNote.js:13-43`

### UPDATE Policy: Edit Notes

**Policy Name:** `notes_update_policy`

**Rule:** Authenticated users can update any note in their organization

```sql
CREATE POLICY "notes_update_policy"
ON public.notes
FOR UPDATE
TO authenticated
USING (
  -- Can only update notes in user's organization
  organization_id = auth.current_organization_id()
)
WITH CHECK (
  -- Organization cannot be changed
  organization_id = auth.current_organization_id()

  -- Parent entities cannot be changed (immutable after creation)
  -- This is enforced at application level, not RLS
);
```

**Update Restrictions:**
1. **Organization Immutable**: Cannot change `organization_id` (WITH CHECK prevents)
2. **Parent Entity Immutable**: Cannot change `customer_id`, `project_id`, or `task_id` (application-level validation)
3. **Allowed Updates**: `note` content, `type` field

**Design Decision:**
- **No "creator-only" restriction**: Matches current FileMaker (no update restrictions)
- Future enhancement: Add `created_by` check if needed
- **Rationale**: Simple permission model, all org users trusted equally

**Frontend Implications:**
- Edit UI can be shown to all users (no per-note permission checks needed)
- Backend API validates update requests, rejects parent entity changes
- `updated_at` and `updated_by` automatically set by trigger

**Code Reference:** `requirements/notes/api-contracts.md:220-289` (Update endpoint spec)

### DELETE Policy: Remove Notes

**Policy Name:** `notes_delete_policy`

**Rule:** Authenticated users can delete any note in their organization

```sql
CREATE POLICY "notes_delete_policy"
ON public.notes
FOR DELETE
TO authenticated
USING (
  -- Can only delete notes in user's organization
  organization_id = auth.current_organization_id()
);
```

**Design Decision:**
- **No "creator-only" restriction**: Matches current FileMaker (simple model)
- Future enhancement: Add `created_by` check or admin role check if needed
- **Hard delete**: Notes are permanently removed (no soft delete)

**Cascade Behavior:**
- Notes are NOT cascaded when parent deleted (ON DELETE CASCADE on parent side)
- Deleting a project/task/customer will cascade delete all associated notes
- This is expected behavior to maintain referential integrity

**Frontend Implications:**
- Delete buttons can be shown to all users
- Confirm deletion via modal/dialog (irreversible operation)
- Backend performs actual deletion

**Future Enhancement Consideration:**
- Soft delete: Add `deleted_at` column, filter deleted notes in queries
- Audit trail: Keep deletion record for compliance

## Parent Entity Access Inheritance

### Access Model

**Principle:** If a user can access a parent entity (customer/project/task), they can create and view notes on that entity.

**Implementation:**
1. **Parent Entity RLS Policies**: Customers, projects, and tasks have their own RLS policies
2. **Notes RLS Policies**: Notes policies check parent entity access via helper functions
3. **Transitive Access**: Note access inherits from parent access

**Example Scenarios:**

#### Scenario 1: Project Notes
```
User A (Org 1) → Can access Project X (Org 1)
  ↓
User A can create/view notes on Project X

User B (Org 2) → Cannot access Project X (Org 1)
  ↓
User B cannot create/view notes on Project X
```

#### Scenario 2: Task Notes
```
User A (Org 1) → Can access Project X (Org 1) → Can access Task Y (Project X)
  ↓
User A can create/view notes on Task Y

Task Z (Project W, Org 2) → User A cannot access Project W
  ↓
User A cannot create notes on Task Z (policy violation via has_task_access())
```

---

### Parent Entity Organization Scoping

**Customers:**
- Table: `public.customers`
- Organization Field: `organization_id` (direct)
- RLS Policy: Users can only access customers in their organization

**Projects:**
- Table: `public.projects`
- Organization Field: `organization_id` (direct)
- RLS Policy: Users can only access projects in their organization

**Tasks:**
- Table: `public.tasks`
- Organization Field: Inherited from `projects.organization_id` (via `project_id` FK)
- RLS Policy: Users can only access tasks whose parent project is in their organization

**Notes:**
- Table: `public.notes`
- Organization Field: `organization_id` (derived from parent entity)
- RLS Policy: Users can only access notes in their organization + parent access validated on INSERT

---

### Orphaned Notes Prevention

**Problem:** What if a parent entity is deleted after note creation?

**Solution:** Cascade deletion on foreign keys

```sql
-- Notes table foreign keys
customer_id UUID REFERENCES public.customers(id) ON DELETE CASCADE,
project_id UUID REFERENCES public.projects(id) ON DELETE CASCADE,
task_id UUID REFERENCES public.tasks(id) ON DELETE CASCADE,
```

**Behavior:**
- Deleting a customer → All customer notes deleted
- Deleting a project → All project notes deleted
- Deleting a task → All task notes deleted

**Rationale:**
- Notes without a parent are meaningless (no context)
- Prevents orphaned records cluttering database
- Matches user expectation (deleting project removes all related data)

**Alternative Considered:**
- `ON DELETE SET NULL` → Creates orphaned notes (bad UX)
- Soft delete on parent → Still allows note access (consider for future)

**Notes:**
- "Accessible entities" = Customers/projects/tasks in user's organization
- All authenticated users have equal permissions (no role-based restrictions)

## Service Role vs Anon Key Usage

### Service Role Key (`VITE_SUPABASE_SERVICE_ROLE_KEY`)

**When to Use:**
- Backend API operations (server-side only)
- Data migrations
- Admin operations
- Bulk operations (seeding data, backfill scripts)

**Characteristics:**
- **Bypasses ALL RLS policies**
- **Full database access** (can read/write any data)
- **NEVER expose to frontend** (critical security risk)
- **Environment:** Backend server environment variables only

**Usage Pattern:**
```javascript
// Backend service
import { createClient } from '@supabase/supabase-js';

const supabaseAdmin = createClient(
  process.env.SUPABASE_URL,
  process.env.SUPABASE_SERVICE_ROLE_KEY  // SERVICE ROLE - Backend only!
);

// Can access all notes across all organizations
const { data } = await supabaseAdmin.from('notes').select('*');
```

**Code Reference:**
- Supabase service: `src/services/supabaseService.js`
- Not currently used in frontend (stays in backend)

---

### Anon Key (`VITE_SUPABASE_ANON_KEY`)

**When to Use:**
- Frontend Supabase client initialization
- Direct database queries from browser
- Realtime subscriptions
- User authentication flows

**Characteristics:**
- **Subject to RLS policies** (organization scoping enforced)
- **Safe for frontend** (can be exposed in client code)
- **Limited by JWT claims** (user's organization_id determines access)
- **Environment:** Frontend `.env` file, safe to expose

**Usage Pattern:**
```javascript
// Frontend service
import { supabaseService } from '@/services/supabaseService';

// Queries automatically scoped by RLS
const { data } = await supabaseService.client
  .from('notes')
  .select('*')
  .eq('project_id', projectId);
// Returns only notes where:
// - project_id = projectId
// - organization_id = user's organization (automatic via RLS)
```

**Code Reference:** `src/services/supabaseService.js`

## Backend API Authorization

### HMAC Authentication Flow

**Request Flow:**
1. Frontend calls backend API: `POST /api/notes`
2. `dataService.generateBackendAuthHeader()` creates HMAC signature
3. Request includes: `Authorization: Bearer {signature}.{timestamp}`
4. Backend validates HMAC signature using shared secret
5. Backend extracts user context from session/JWT
6. Backend queries Supabase using **service role key** (bypasses RLS)
7. Backend enforces organization scoping at application level

**Code Reference:**
- Frontend: `src/services/dataService.js:generateBackendAuthHeader()`
- Backend: (Handled by backend team - not in frontend codebase)

---

### Backend Organization Scoping

**Backend Responsibility:**
Even though service role bypasses RLS, backend MUST enforce organization scoping:

```javascript
// Backend API pseudo-code
app.post('/api/notes', async (req, res) => {
  // 1. Validate HMAC signature
  if (!validateHMAC(req.headers.authorization)) {
    return res.status(401).json({ error: 'Invalid authentication' });
  }

  // 2. Get user context (from session, JWT, etc.)
  const user = await getUserFromSession(req);
  const userOrgId = user.organization_id;

  // 3. Validate parent entity belongs to user's organization
  const { project_id } = req.body;
  const project = await supabaseAdmin
    .from('projects')
    .select('organization_id')
    .eq('id', project_id)
    .single();

  if (project.data.organization_id !== userOrgId) {
    return res.status(403).json({ error: 'Access denied' });
  }

  // 4. Create note with user's organization_id
  const { data, error } = await supabaseAdmin
    .from('notes')
    .insert({
      ...req.body,
      organization_id: userOrgId,  // CRITICAL: Set from user context
      created_by: user.id
    });

  return res.json({ success: true, data });
});
```

**Critical Points:**
- Backend MUST NOT trust `organization_id` from frontend request body
- Backend derives `organization_id` from authenticated user context
- Backend validates parent entity access before creating note
- Backend uses service role but implements application-level authorization

**Layer 1: Frontend Validation** (`src/services/noteService.js:11-17`)

## Security Considerations

### Multi-Tenancy Isolation

**Requirement:** ABSOLUTE organization isolation - no cross-org data leaks

**Enforcement Mechanisms:**
1. **RLS Policies**: Automatically filter all queries by `organization_id`
2. **Backend Validation**: Application-level organization checks
3. **JWT Claims**: User's organization embedded in authentication token
4. **Database Indexes**: Efficient organization filtering

**Testing:**
- Create notes in multiple organizations
- Attempt cross-org access with manipulated requests
- Verify RLS policies deny unauthorized access
- Test with multiple concurrent users from different orgs

**Code Reference:** Test cases to be added in `requirements/notes/acceptance-criteria.md`

---

### Parent Entity Access Control

**Requirement:** Users cannot create notes on entities they don't have access to

**Enforcement:**
1. **INSERT Policy**: Calls `has_customer_access()`, `has_project_access()`, `has_task_access()`
2. **Helper Functions**: Check parent entity organization membership
3. **Foreign Key Constraints**: Ensure parent entity exists

**Attack Scenarios:**
- **Scenario 1**: User tries to create note on another org's project
  - **Defense**: `has_project_access()` returns false, INSERT policy denies
- **Scenario 2**: User manipulates `organization_id` in request
  - **Defense**: Backend derives org_id from user context, ignores request value
- **Scenario 3**: User tries to create note with non-existent parent
  - **Defense**: Foreign key constraint violation before RLS check

---

### XSS Prevention

**Requirement:** Note content may contain user-generated text, prevent XSS attacks

**Mitigation:**
1. **Frontend Display**: Escape HTML when rendering note content
   ```javascript
   // Use React's built-in XSS protection
   <div>{note.content}</div>  // Automatically escaped
   ```
2. **Backend Storage**: Store as plain text, no HTML injection
3. **Future Rich Text**: If implementing rich text, use sanitization library (DOMPurify)

**Code Reference:**
- Display: `src/components/projects/ProjectNotesTab.jsx`
- Display: `src/components/tasks/TaskList.jsx:108-131`

---

### SQL Injection Prevention

**Requirement:** Prevent SQL injection via note content or query parameters

**Mitigation:**
1. **Parameterized Queries**: Supabase client uses prepared statements
   ```javascript
   // Safe - parameters are escaped
   .eq('project_id', projectId)
   .insert({ note: userInput })
   ```
2. **RLS Policies**: Use SQL parameters, not string concatenation
3. **Backend API**: Use ORM or parameterized queries

**No Risk From:**
- Note content stored as TEXT (not executed as SQL)
- Supabase client API (handles parameterization)
- RLS helper functions (use parameters, not string concat)

---

### CSRF Protection

**Backend API CSRF Protection:**
- HMAC signature includes timestamp
- Signature cannot be reused (timestamp freshness check)
- Each request requires fresh signature generation
- Prevents replay attacks

**Supabase Direct Access:**
- JWT tokens tied to user session
- SameSite cookie policies prevent CSRF
- Origin headers validated by Supabase

**Code Reference:** `src/services/dataService.js:generateBackendAuthHeader()`

---

### Rate Limiting

**Recommendation:** Backend should implement rate limiting

**Suggested Limits:**
- **Per User:**
  - 100 requests per minute (prevent abuse)
  - 500 notes created per day (prevent spam)
- **Per Organization:**
  - 10,000 notes total (alert if exceeded, possible data issue)

**Implementation:** Backend team responsibility (not frontend)

**Monitoring:**
- Track note creation rate by user/organization
- Alert on suspicious patterns (bulk creation, rapid deletion)

## Summary

**Authorization Model:**
- ✅ Organization-scoped access (all authenticated users in org have equal permissions)
- ✅ Parent entity access inheritance (can create notes on accessible entities)
- ✅ Simple permission model (matches current FileMaker behavior)
- ✅ No role-based restrictions (all org users can create/view/edit/delete)

**Security Enforcement:**
- ✅ RLS policies for Supabase direct access (anon key queries)
- ✅ Application-level checks for Backend API (service role queries)
- ✅ Multi-layered validation (frontend, backend, database)
- ✅ Cascade deletion prevents orphaned notes

**Key Design Decisions:**
- ✅ Simple over complex: Start with basic permissions, enhance later if needed
- ✅ Security by default: RLS enforced at database level
- ✅ Performance optimized: Indexed organization_id for efficient filtering
- ✅ Developer friendly: Clear helper functions, well-documented policies

**Ready for Implementation:** Backend team can implement based on this specification.
