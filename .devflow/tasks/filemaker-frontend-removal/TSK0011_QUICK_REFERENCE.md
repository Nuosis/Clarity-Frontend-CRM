# TSK0011 Quick Reference - DualWriteService Deprecation

**Status:** ✅ Deprecated (Not Removed)
**Date:** 2026-01-15

---

## TL;DR

**DO NOT USE `dualWriteService.js`** - It's deprecated. Use backend API endpoints directly via `dataService` or feature-specific API clients.

---

## What Happened?

The `dualWriteService` was designed to coordinate writes between FileMaker (primary) and Supabase (secondary) during migration. Now that backend API integration is complete (TSK0004-TSK0006), this dual-write pattern is **obsolete**.

**Key Facts:**
- ✅ Service never used in production code (verified)
- ✅ Backend API now handles all data persistence
- ✅ Timer operations use backend `/time-entries` endpoints
- ✅ Financial records auto-generated by backend
- ✅ No dual-write coordination needed

---

## Migration Cheat Sheet

### Timer Operations

```javascript
// ❌ DON'T USE
import { stopTimerWithDualWrite } from '../services/dualWriteService';
await stopTimerWithDualWrite(params, orgId);

// ✅ USE THIS
import { stopTaskTimer } from '../api/tasks';
await stopTaskTimer(params);
// Backend automatically creates financial records & sales records
```

### Data Operations

```javascript
// ❌ DON'T USE
import { withDualWrite } from '../services/dualWriteService';
await withDualWrite(
  async () => await fileMakerOperation(),
  { operationType: 'record_create', organizationId, recordData }
);

// ✅ USE THIS
import { dataService } from '../services/dataService';
await dataService.post('devCustomers', data);
// Environment-aware: backend API primary, FileMaker fallback
```

### Sales Records

```javascript
// ❌ DON'T USE (Manual dual-write)
import { createSaleFromFinancialRecord } from '../services/salesService';
if (timerResult.success) {
  await createSaleFromFinancialRecord(financialId, orgId);
}

// ✅ USE THIS (Automatic)
import { stopTaskTimer } from '../api/tasks';
await stopTaskTimer(params);
// Sales records created automatically by backend
```

## Current Architecture

### Timer Lifecycle (Post-Migration)

```
1. User stops timer
   ↓
2. Frontend: stopTaskTimer() → Backend API
   ↓
3. Backend: Stops timer & creates financial record
   ↓
4. Backend: Creates customer_sales record (if not fixed-price)
   ↓
5. Frontend: Receives success response
```

**No Frontend Dual-Write Required** - Backend handles everything.

### Data Flow Pattern

```
Frontend Component
   ↓
API Client (src/api/*.js)
   ↓
dataService (environment-aware routing)
   ↓
Backend API (primary) OR FileMaker (legacy fallback)
   ↓
Database
```

**Feature Flags** control which path is used (`use_backend_customers`, etc.)
