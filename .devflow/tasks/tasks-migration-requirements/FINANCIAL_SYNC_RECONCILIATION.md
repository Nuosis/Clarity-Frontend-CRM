# Financial Sync Reconciliation Documentation

## Overview

The `financialSyncService.js` provides synchronization between FileMaker `devRecords` (source of truth) and Supabase `customer_sales` table. This service ensures that financial records generated by timer operations are accurately reflected in the Supabase database for QuickBooks integration and financial reporting.

**Key Files:**
- `src/services/financialSyncService.js` - Main synchronization service
- `src/services/syncTrackingService.js` - Session-based sync tracking
- `src/services/billableHoursService.js` - Financial data processing
- `src/api/financialRecords.js` - FileMaker devRecords API

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                    Synchronization Flow                          │
└─────────────────────────────────────────────────────────────────┘

1. Fetch devRecords (FileMaker - Source of Truth)
   └─> fetchRecordsForDateRange(startDate, endDate)
       └─> FileMaker devRecords layout
           └─> processFinancialData() transforms to standard format

2. Fetch customer_sales (Supabase - Target)
   └─> fetchCustomerSalesForDateRange(organizationId, startDate, endDate)
       └─> Supabase customer_sales table
           └─> Filters: organization_id, date range

3. Compare Records
   └─> compareRecords(devRecords, customerSales, organizationId)
       ├─> Match by financial_id (case-insensitive)
       ├─> Identify differences (toCreate, toUpdate, toDelete, unchanged)
       └─> Store pending operations (sessionStorage)

4. Apply Synchronization (Optional)
   ├─> Create missing customer_sales records
   ├─> Update changed customer_sales records
   └─> Delete orphaned customer_sales records (optional)
```

## Core Functions

### 1. `synchronizeFinancialRecords()`

**Location:** `src/services/financialSyncService.js:25-260`

Main synchronization function that orchestrates the entire sync process.

**Parameters:**
```javascript
{
  organizationId: string,        // Organization UUID
  startDate: string,             // YYYY-MM-DD format
  endDate: string,               // YYYY-MM-DD format
  options: {
    dryRun: boolean,            // If true, only reports changes without applying (default: false)
    deleteOrphaned: boolean,    // If true, deletes customer_sales not in devRecords (default: false)
    usePendingOnly: boolean     // If true, uses stored sync tracking instead of full review (default: false)
  }
}
```

**Return Value:**
```javascript
{
  success: boolean,
  summary: {
    devRecordsCount: number,
    customerSalesCount: number,
    toCreate: number,            // Records needing creation
    toUpdate: number,            // Records needing updates
    toDelete: number,            // Orphaned records (if deleteOrphaned=true)
    unchanged: number,           // Records already in sync
    syncType: 'full_review' | 'pending_only'
  },
  changes: {
    created: Array,              // Successfully created records
    updated: Array,              // Successfully updated records
    deleted: Array,              // Successfully deleted records
    errors: Array                // Errors encountered during sync
  },
  processedRecordIds: Array,     // All processed devRecord IDs
  duration: number,              // Sync duration in milliseconds
  dryRun: boolean
}
```

**Execution Modes:**

1. **Full Review Mode** (`usePendingOnly: false`)
   - Fetches all devRecords and customer_sales for date range
   - Performs complete comparison
   - Stores pending operations in sessionStorage
   - Applies changes (if not dryRun)

2. **Pending Only Mode** (`usePendingOnly: true`)
   - Uses previously stored sync tracking from sessionStorage
   - Skips fetching and comparison
   - Directly applies pending changes
   - Useful for "Review → Approve → Sync" workflows

### 4. `compareRecords()`

**Location:** `src/services/financialSyncService.js:345-400`

Compares devRecords with customer_sales to identify differences.

**Matching Logic:**
```javascript
// Build lookup map (case-insensitive)
const customerSalesMap = new Map();
customerSales.forEach(sale => {
  if (sale.financial_id) {
    // Store with lowercase key for case-insensitive matching
    customerSalesMap.set(sale.financial_id.toLowerCase(), sale);
  }
});

// Check each devRecord
for (const devRecord of devRecords) {
  const existingSale = customerSalesMap.get(devRecord.id.toLowerCase());

  if (!existingSale) {
    // Record doesn't exist → toCreate
    toCreate.push(devRecord);
  } else {
    // Record exists → check for changes
    const changes = await identifyChanges(devRecord, existingSale, organizationId);
    if (Object.keys(changes).length > 0) {
      toUpdate.push({ devRecord, customerSale: existingSale, changes });
    } else {
      unchanged.push({ devRecord, customerSale: existingSale });
    }
    // Remove from map to identify orphaned records
    customerSalesMap.delete(devRecord.id.toLowerCase());
  }
}

// Remaining records are orphaned (exist in customer_sales but not devRecords)
customerSalesMap.forEach(sale => toDelete.push(sale));
```

**Return Value:**
```javascript
{
  toCreate: Array<devRecord>,              // Records to insert
  toUpdate: Array<{                        // Records to update
    devRecord: Object,
    customerSale: Object,
    changes: Object
  }>,
  toDelete: Array<customerSale>,           // Orphaned records
  unchanged: Array<{                       // Records already in sync
    devRecord: Object,
    customerSale: Object
  }>
}
```

### 5. `identifyChanges()`

**Location:** `src/services/financialSyncService.js:409-453`

Detects what fields need updating in a customer_sales record.

**Comparison Logic:**
```javascript
const changes = {};

// 1. Check quantity (hours) - rounded to 2 decimal places
const devHours = Math.round(Number(devRecord.hours) * 100) / 100;
const saleQuantity = Math.round(Number(customerSale.quantity) * 100) / 100;
if (devHours !== saleQuantity) {
  changes.quantity = Number(devRecord.hours);
}

// 2. Check unit_price (rate) - rounded to 2 decimal places
const devRate = Math.round(Number(devRecord.rate) * 100) / 100;
const saleUnitPrice = Math.round(Number(customerSale.unit_price) * 100) / 100;
if (devRate !== saleUnitPrice) {
  changes.unit_price = Number(devRecord.rate);
}

// 3. Check total_price (amount) - rounded to 2 decimal places
const devAmount = Math.round(Number(devRecord.amount) * 100) / 100;
const saleTotalPrice = Math.round(Number(customerSale.total_price) * 100) / 100;
if (devAmount !== saleTotalPrice) {
  changes.total_price = Number(devRecord.amount);
}

// 4. Check date (YYYY-MM-DD format)
const devRecordDate = new Date(devRecord.date).toISOString().split('T')[0];
const customerSaleDate = new Date(customerSale.date).toISOString().split('T')[0];
if (devRecordDate !== customerSaleDate) {
  changes.date = devRecordDate;
}

// 5. Check product_name (formatted: "CBS:Website")
const expectedProductName = formatProductName(devRecord.customerName, devRecord.projectName);
if (expectedProductName !== customerSale.product_name) {
  changes.product_name = expectedProductName;
}

// 6. Check customer_id (ensure it matches customer name)
const expectedCustomerId = await getOrCreateCustomerId(devRecord.customerName, organizationId);
if (expectedCustomerId !== customerSale.customer_id) {
  changes.customer_id = expectedCustomerId;
}

return changes; // Returns empty object if no changes detected
```

**Precision Handling:**
- All numeric comparisons use 2 decimal place rounding to avoid floating point issues
- Dates normalized to YYYY-MM-DD format for comparison

**Fields Checked:**
1. `quantity` (hours)
2. `unit_price` (rate)
3. `total_price` (amount)
4. `date`
5. `product_name`
6. `customer_id`

**Product Name Formatting:**

**Function:** `formatProductName()` (line 628-640)

```javascript
function formatProductName(customerName, projectName) {
  // Extract capital letters and numbers from customer name
  const customerNameFormatted = (customerName || '')
    .replace(/[^A-Z0-9]/g, '')  // Keep only capital letters and numbers
    .trim();

  // Get the first word of the project name
  const projectNameFirstWord = projectName ?
    projectName.split(' ')[0] : '';

  // Concatenate with a colon
  return `${customerNameFormatted}:${projectNameFirstWord}`;
}
```

**Examples:**
- Customer: "Clarity Business Solutions", Project: "Website Redesign" → "CBS:Website"
- Customer: "ABC Corp", Project: "Database Migration" → "ABC:Database"
- Customer: "123 Industries", Project: "API Development" → "123:API"

## Reconciliation Strategies

### 1. **Full Review and Sync** (Recommended for Manual Reviews)

```javascript
// Step 1: Review what needs syncing (dry run)
const reviewResult = await synchronizeFinancialRecords(
  organizationId,
  startDate,
  endDate,
  { dryRun: true }
);

console.log('Review Summary:', reviewResult.summary);
console.log('Changes to apply:', reviewResult.changes);

// Step 2: Apply changes if approved
if (userApproved) {
  const syncResult = await synchronizeFinancialRecords(
    organizationId,
    startDate,
    endDate,
    { dryRun: false }
  );
  console.log('Sync completed:', syncResult);
}
```

**Use Cases:**
- Manual monthly reconciliation
- One-time historical data sync
- Investigation of discrepancies

### 2. **Review → Approve → Sync** (Recommended for UI Workflows)

```javascript
// Step 1: Full review and store pending operations
const reviewResult = await synchronizeFinancialRecords(
  organizationId,
  startDate,
  endDate,
  { dryRun: false, usePendingOnly: false }
);

// Pending operations now stored in sessionStorage

// Step 2: User reviews summary
const pendingSummary = getPendingSyncSummary(organizationId, startDate, endDate);
console.log('Pending operations:', pendingSummary);

// Step 3: User approves and executes pending sync
if (userApproved) {
  const syncResult = await synchronizeFinancialRecords(
    organizationId,
    startDate,
    endDate,
    { dryRun: false, usePendingOnly: true }
  );

  // Clear tracking after successful sync
  clearSyncTracking(organizationId, startDate, endDate);
}
```

**Advantages:**
- Separates review from execution
- User can examine changes before applying
- Survives page refreshes within session
- No need to re-fetch data for execution

### 3. **Orphaned Record Handling**

```javascript
// Sync with orphaned record deletion
const syncResult = await synchronizeFinancialRecords(
  organizationId,
  startDate,
  endDate,
  {
    dryRun: false,
    deleteOrphaned: true  // Delete customer_sales not in devRecords
  }
);
```

**Warning:** Use with caution. Orphaned records may indicate:
- Records manually created in customer_sales
- Records from other systems
- Data migration artifacts

**Recommendation:** Default to `deleteOrphaned: false` and investigate orphaned records manually.

### 4. **Status Check Without Sync**

## Error Handling

### Create Errors

```javascript
syncResult.changes.errors.push({
  type: 'create',
  devRecordId: devRecord.id,
  error: error.message
});
```

**Common Causes:**
- Customer lookup/creation failed
- Supabase insert constraint violation
- Network issues
- Invalid data types

### Update Errors

```javascript
syncResult.changes.errors.push({
  type: 'update',
  customerSaleId: customerSale.id,
  devRecordId: devRecord.id,
  error: error.message
});
```

**Common Causes:**
- Customer_sales record doesn't exist (deleted externally)
- Concurrent update conflicts
- Network issues

### Delete Errors

```javascript
syncResult.changes.errors.push({
  type: 'delete',
  customerSaleId: customerSale.id,
  error: error.message
});
```

**Common Causes:**
- Foreign key constraints (record referenced elsewhere)
- Record already deleted
- Network issues

### Error Recovery

**Partial Failures:**
- Sync continues even if individual operations fail
- All errors collected in `syncResult.changes.errors`
- Successful operations tracked in `created`, `updated`, `deleted` arrays

**Retry Strategy:**
```javascript
// After reviewing errors, retry failed operations
const failedRecordIds = syncResult.changes.errors
  .filter(e => e.type === 'create')
  .map(e => e.devRecordId);

// Re-run sync for specific records (requires custom implementation)
```

**Current Limitation:** No automatic retry mechanism implemented. Manual intervention required for failed operations.

## Data Consistency Considerations

### 1. **Source of Truth**

FileMaker `devRecords` is the **source of truth** for financial data:
- Timer operations write to FileMaker first
- Supabase `customer_sales` is a **derived dataset** for QuickBooks integration
- Conflicts always resolved in favor of devRecords

### 2. **Case-Insensitive Matching**

```javascript
// financial_id matching is case-insensitive
customerSalesMap.set(sale.financial_id.toLowerCase(), sale);
const existingSale = customerSalesMap.get(devRecord.id.toLowerCase());
```

**Rationale:** FileMaker UUIDs may have inconsistent casing across systems.

### 3. **Floating Point Precision**

All numeric comparisons use 2 decimal place rounding:

```javascript
const devHours = Math.round(Number(devRecord.hours) * 100) / 100;
const saleQuantity = Math.round(Number(customerSale.quantity) * 100) / 100;
```

**Rationale:** Prevents spurious updates due to floating point representation differences.

### 4. **Date Normalization**

All dates converted to YYYY-MM-DD format for comparison:

```javascript
const devRecordDate = new Date(devRecord.date).toISOString().split('T')[0];
const customerSaleDate = new Date(customerSale.date).toISOString().split('T')[0];
```

**Rationale:** FileMaker and Supabase may store dates in different formats.

### 5. **Customer Name Matching**

Customers matched by exact `business_name`:

```javascript
query('customers', {
  eq: { column: 'business_name', value: customerName }
});
```

**Potential Issue:** Customer name changes in FileMaker won't automatically update Supabase linkage.

**Recommendation:** Implement customer_id mapping table for FileMaker ↔ Supabase reconciliation.

## Fixed-Price Project Handling

**Important Context:** Timer stop operations check `customers_Projects::f_fixedPrice`:
- If `f_fixedPrice > 0`: No customer_sales record created (see `taskService.js:108-112`)
- If `f_fixedPrice = 0`: Normal customer_sales record created

**Impact on Sync:**
- Fixed-price project hours will appear in devRecords
- They should NOT appear in customer_sales
- Sync service will flag them as "toCreate"

**Current Behavior:**
- Sync service **does not check** `fixedPrice` field
- Will attempt to create customer_sales records for fixed-price projects

**Recommendation:**
```javascript
// In compareRecords(), filter out fixed-price records
for (const devRecord of devRecords) {
  // Skip fixed-price project records
  if (devRecord.fixedPrice && devRecord.fixedPrice > 0) {
    console.log(`Skipping fixed-price record: ${devRecord.id}`);
    continue;
  }

  // Normal matching logic...
}
```

**Migration Note:** This filtering logic should be added to prevent sync service from creating unwanted customer_sales records for fixed-price projects.

## Migration Considerations for Supabase

### Current State
- **FileMaker devRecords:** Source of truth for timer data
- **Supabase customer_sales:** Derived dataset for QuickBooks integration
- **Sync Direction:** One-way from FileMaker → Supabase

### Future State (Full Supabase Migration)

When migrating to Supabase-native timer system:

1. **Replace devRecords with time_entries table**
   - Proposed schema documented in `DEVRECORDS_SUPABASE_MAPPING.md`
   - time_entries becomes source of truth
   - customer_sales derived from time_entries

2. **Update Sync Service**
   - Change `fetchDevRecordsForDateRange()` to query `time_entries`
   - Update matching logic to use time_entries.id instead of financial_id
   - Add organization_id filtering to time_entries queries

3. **Backward Compatibility**
   - Maintain FileMaker sync during transition period
   - Dual-write to both devRecords and time_entries
   - Gradually phase out FileMaker dependency

4. **Data Migration**
   - One-time migration of historical devRecords to time_entries
   - Reconcile customer_sales with migrated time_entries
   - Verify financial_id linkages

### Recommended Migration Phases

**Phase 1: Shadow Mode (Current)**
- FileMaker devRecords = source of truth
- customer_sales synced from devRecords
- financialSyncService.js operates as designed

**Phase 2: Dual Write**

## Summary

The financial sync reconciliation service provides:

1. **Automated Synchronization:** Ensures customer_sales matches devRecords
2. **Change Detection:** Identifies creates, updates, deletes with field-level comparison
3. **Flexible Execution:** Supports dry-run, review-then-sync, and direct sync workflows
4. **Error Tracking:** Collects all errors for manual review and retry
5. **Session Tracking:** Uses sessionStorage for review → approve workflows
6. **Customer Management:** Automatically creates missing customers and organization links
7. **Product Name Formatting:** Standardizes product names for consistency

**Key Design Decisions:**
- FileMaker devRecords is source of truth
- Case-insensitive financial_id matching
- 2 decimal place precision for numeric comparisons
- Date normalization to YYYY-MM-DD
- Full field overwrite on updates (not delta updates)
- No automatic retry on failures
- No fixed-price project filtering (current limitation)
