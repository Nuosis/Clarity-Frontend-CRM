# Task CRUD Operations Documentation

## Overview

This document provides a comprehensive overview of task Create, Read, Update, Delete (CRUD) operations in the Clarity CRM Frontend application. Tasks are currently stored in FileMaker and accessed via the fm-gofer bridge. There is NO Supabase tasks table for customer tasks (only task scheduling/automation tables exist).

**Current State:** FileMaker-only implementation (LEGACY)
**Migration Target:** Supabase with backend API integration

### Layouts

| Layout Name | Purpose | Used For |

## Validation Rules

### TASK_FIELDS Validation Schema

Defined in src/services/taskService.js:265-305:

```javascript
export const TASK_FIELDS = {
    task: {
        required: true,
        type: 'string',
        minLength: 1,
        maxLength: 200,
        validate: (value) => {
            if (!value?.trim()) return 'Task name is required';
            if (value.length > 200) return 'Task name must be less than 200 characters';
            return null;
        }
    },
    _projectID: {
        required: true,
        type: 'string',
        validate: (value) => !value ? 'Project ID is required' : null
    },
    _staffID: {
        required: true,
        type: 'string',
        validate: (value) => !value ? 'Staff ID is required' : null
    },
    f_completed: {
        required: false,
        type: 'number',
        validate: (value) => {
            if (value === undefined || value === null) return null;
            return value === 0 || value === 1 ? null : 'Completion status must be 0 or 1';
        }
    },
    f_priority: {
        required: false,
        type: 'string',
        validate: (value) => {
            if (!value) return null;
            return ['active', 'high', 'low'].includes(value) ? null : 'Invalid priority value';
        }
    }
};
```

### Validation Function

`validateTaskData(data, options)` in src/services/taskService.js:315-376:

**Parameters:**
- `data` (Object): Task data to validate
- `options.isUpdate` (boolean): Whether this is an update operation
- `options.partial` (boolean): Whether to allow partial data

**Returns:**
```javascript
{
    isValid: boolean,
    errors: string[],
    fieldErrors: { [field]: string }
}
```

**Validation Logic:**
1. Checks data is a valid object
2. Validates each field according to TASK_FIELDS rules
3. Skips required field validation for updates and partial validation
4. Skips `_projectID` validation for updates (can't change project)
5. Runs type checking and custom validators

## CRUD Operations

### Create Task

**Service Layer:** `createNewTask(params)` in src/services/taskService.js:414-467

**API Layer:** `createTask(data)` in src/api/tasks.js:46-70

**Flow:**
1. Extract parameters (supports both old and new field name formats)
2. Validate required fields: `projectId`, `staffId`, `taskName`
3. Generate UUID for `__ID` field
4. Prepare task data object:
   ```javascript
   {
       __ID: uuidv4(),
       _projectID: projectId,
       _staffID: staffId,
       task: taskName.trim(),
       f_completed: 0,
       f_priority: priority || "active"
   }
   ```
5. Validate task data using `validateTaskData()`
6. Call FileMaker CREATE action on `devTasks` layout
7. Return formatted response with recordId and UUID

**FileMaker Request:**
```javascript
{
    layout: 'devTasks',
    action: 'CREATE',
    fieldData: {
        __ID: 'generated-uuid',
        _projectID: 'project-id',
        _staffID: 'staff-id',
        task: 'Task name',
        f_completed: 0,
        f_priority: 'active'
    }
}
```

**Response:**
```javascript
{
    response: {
        recordId: '12345',
        modId: '1',
        data: [{
            recordId: '12345',
            modId: '1',
            fieldData: { /* task data with UUID */ }
        }]
    },
    messages: [{ code: '0', message: 'OK' }]
}
```

**Hook Usage:** `handleTaskCreate(taskData)` in src/hooks/useTask.js:107-144

## Timer Operations

### Start Timer

**Service Layer:** `startTimer(task)` in src/services/taskService.js:55-60

**API Layer:** `startTaskTimer(taskId, selectedTask)` in src/api/tasks.js:121-179

**Flow:**
1. Validate task has an ID
2. Fetch project details to get customer ID (`_custID`)
   - First tries to find project by `__ID`
   - Falls back to finding by `recordId`
3. Create timer record on `dapiRecords` layout with:
   - Task, staff, project, and customer IDs
   - Start time (HH:MM:SS format, 24-hour)
   - Start date (MM/DD/YYYY format)
4. Store timer record ID in local state and localStorage

**FileMaker Request:**
```javascript
{
    layout: 'dapiRecords',
    action: 'CREATE',
    fieldData: {
        _taskID: 'task-id',
        _staffID: 'staff-id',
        _projectID: 'project-id',
        _custID: 'customer-id',
        TimeStart: '14:30:00',
        DateStart: '01/10/2026'
    }
}
```

**Hook Usage:** `handleTimerStart(task)` in src/hooks/useTask.js:204-235

**Timer State:**
```javascript
{
    recordId: 'timer-record-id',
    TimeStart: '14:30:00',
    isPaused: false,
    pauseStartTime: null,
    totalPauseTime: 0,
    adjustment: 0
}
```

Stored in localStorage as `activeTimer`.

### Stop Timer

**Service Layer:** `stopTimer(params, organizationId)` in src/services/taskService.js:67-131

**API Layer:** `stopTaskTimer(recordId, description, saveImmediately, TimeAdjust)` in src/api/tasks.js:189-213

**Flow:**
1. Validate timer record ID exists
2. Update timer record on `dapiRecords` layout with:
   - End time (current time in HH:MM:SS format)
   - Work description (or 'Time logged' if saved immediately)
   - Time adjustment (total pause time + manual adjustment in seconds)
3. Fetch financial record by recordId to get financial ID
4. Check if project is fixed-price (`f_fixedPrice > 0`)
5. If NOT fixed-price, create sales record in Supabase via `createSaleFromFinancialRecord()`
6. Clear timer state from localStorage

**FileMaker Request:**
```javascript
{
    layout: 'dapiRecords',
    action: 'UPDATE',
    recordId: 'timer-record-id',
    fieldData: {
        TimeEnd: '16:45:00',
        'Work Performed': 'Work description',
        TimeAdjust: 360  // seconds
    }
}
```

**Fixed-Price Logic:** src/services/taskService.js:104-111

If `customers_Projects::f_fixedPrice > 0`, the timer is stopped but NO sales record is created.

**Hook Usage:** `handleTimerStop(saveImmediately, description)` in src/hooks/useTask.js:237-277

**Hook Function:** `handleTimerPause()` in src/hooks/useTask.js:279-301

**Note:** Pause logic is handled entirely in frontend. FileMaker receives total pause time when timer is stopped via `TimeAdjust` field.

## Migration Considerations

### Current Limitations

1. **FileMaker-Only:** Tasks are ONLY stored in FileMaker
2. **No Supabase Integration:** No tasks table exists in Supabase (only automation tables)
3. **No Dual-Write:** Unlike customers/projects, tasks are not synchronized to Supabase
4. **No Backend API:** No backend API endpoints for tasks
5. **No Delete Operation:** Tasks can only be marked completed, not deleted

### Required for Migration

1. **Backend Changes Needed:**
   - Create Supabase `tasks` table with appropriate schema
   - Create Supabase `task_timers` table for time tracking
   - Create backend API endpoints for task CRUD
   - Create backend API endpoints for timer CRUD
   - Implement financial record generation logic in backend
   - Implement fixed-price project detection

2. **Frontend Changes Needed:**
   - Add Supabase service calls for tasks
   - Implement dual-write pattern (FileMaker + Supabase)
   - Update task API layer to route to backend
   - Add error handling for Supabase operations
   - Update validation for backend requirements

3. **Data Migration:**
   - Export existing tasks from FileMaker
   - Import to Supabase tasks table
   - Verify data integrity
   - Test CRUD operations

4. **Business Logic Considerations:**
   - Fixed-price project logic must be preserved
   - Timer adjustment validation (6-minute increments)
   - Sales record generation on timer stop
   - Customer ID lookup from project
